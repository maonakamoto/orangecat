# Genesis — the Experience Engine

**Status**: Concept / vision capture
**Captured**: 2026-06-13
**Working name**: "Genesis" (provisional — also considered: Muse, Conjure, Oracle)
**Relationship to OrangeCat**: candidate flagship experience layer on top of the OrangeCat economic layer; built/operated via FleetRunner (local) + FleetCrown (web portal).

---

## 1. The one-sentence idea

**You say something, and you get that experience.** Anything — a painting, a piece of music, a poem or short story, a film, an interactive 3D / VR / AR world. One natural-language wish in; a finished, deliverable creative artifact out.

The interface is maximally simple: a single input. No medium picker, no parameter sliders up front. The system infers the medium (or media) from the wish itself and produces it. Complexity is hidden behind progressive disclosure — you can refine, fork, or go deeper after the first result.

## 2. Two ideas, not one

The pitch bundles two distinct capabilities. Worth separating them:

### (a) Universal generation

Prompt → finished artifact, across any modality:

| Modality          | Example wish                                                                        |
| ----------------- | ----------------------------------------------------------------------------------- |
| Image / painting  | "a foggy cedar forest in the style of a faded woodblock print"                      |
| Music             | "a slow ambient piece that feels like being lost and at peace with it"              |
| Text / literature | "a 1,200-word short story about a lighthouse keeper who stops believing in the sea" |
| Film / video      | "a 90-second silent film of a city waking up, shot like a 1970s documentary"        |
| 3D / VR / AR      | "a walkable misty forest clearing I can stand inside with a headset"                |

This is an **orchestration** problem, not a single-model problem: route each wish to the right generative model(s), compose multi-modal outputs, and return something finished.

### (b) Multiverse variants + differential distribution

Generate **many versions of the same work, each differing in a small, deliberate way**, then distribute different versions to different audiences. Some people experience variant A, others variant B. The "work" becomes a branching space rather than a single fixed artifact.

This is the genuinely novel part. Applications:

- **A/B/…/Z storytelling** — narrative forks at scale; the audience is collectively exploring a possibility space.
- **Personalized cuts** — the same film tuned per viewer (pacing, ending, focus character).
- **Living works** — a piece that is never the same twice.
- **Collective experiments** — measure how a tiny change in an artwork changes how people respond.

## 3. Hard constraint: originality & rights (read before building (b))

The variant/distribution mechanic only works on content the platform has the right to transform and redistribute. That means:

- ✅ **Original works** generated by the engine itself.
- ✅ **User-owned works** the user uploads and holds rights to.
- ✅ **Properly licensed / public-domain** source material.
- ❌ **Copyrighted works** (e.g. existing commercial films, their characters, music) used as seeds to produce distributed derivatives. Generating "the same film with a tiny part different" from a protected work and disseminating it is creating and distributing **unauthorized derivative works** — a legal non-starter regardless of how small the change is, and the "tiny change" framing makes it worse, not better.

**Design implication:** the multiverse mechanic is a first-class feature for _original IP_ — worlds, characters, and stories created in (or brought into) the platform with clear provenance. Provenance and rights tracking is therefore part of the data model, not an afterthought. (This also dovetails with OrangeCat's identity/economic layer: rights, attribution, and payout all hang off the same provenance graph.)

## 4. Why this fits OrangeCat + FleetCrown + FleetRunner

The three existing pieces map cleanly onto the product:

```
┌─────────────────────────────────────────────────────────────┐
│  FleetCrown (web portal)        the experience + audience     │
│  · single-input "wish" UI                                     │
│  · browse / fork / refine experiences                         │
│  · audience-facing distribution (who sees which variant)      │
│  · creator dashboard, analytics on variants                   │
├─────────────────────────────────────────────────────────────┤
│  FleetRunner (local Electron app)   the generation engine     │
│  · orchestrates model calls (image/audio/text/video/3D)       │
│  · heavy/local compute, long-running renders                  │
│  · already an agent-fleet runner → each modality = an agent   │
│  · returns finished artifacts + variant trees to the portal   │
├─────────────────────────────────────────────────────────────┤
│  OrangeCat (economic layer)     value, rights, payout         │
│  · provenance + rights graph (who made / owns / licensed what)│
│  · payments: pay-to-generate, pay-to-experience, creator payout│
│  · Bitcoin/Lightning micropayments per experience or variant  │
│  · governance over shared/collective works                    │
└─────────────────────────────────────────────────────────────┘
```

- **FleetCrown** already does "users sign in, register projects, launch/monitor agents from one dashboard." A _wish_ is just a new kind of job; an _experience_ is its output. The audience/variant-distribution surface is new.
- **FleetRunner** is already an Electron agent runner — natural home for the generation orchestration (route a wish to the right model agents, compose, render). Local app = heavy compute, model keys, long renders without a server timeout.
- **OrangeCat** supplies identity, the rights/provenance graph, and money (generation costs, experience access, creator payouts) — the things that turn "a cool generator" into a participatory economy.

## 5. The core loop (MVP shape)

1. **Wish** — user types one sentence in FleetCrown.
2. **Route** — classifier infers modality(ies) + a generation plan.
3. **Generate** — FleetRunner runs the plan (one or more model agents), returns a finished artifact.
4. **Experience** — artifact rendered back in the portal (view/listen/read/play).
5. _(opt.)_ **Fork** — "give me ten variants where X differs" → a variant tree.
6. _(opt.)_ **Distribute** — assign variants to an audience; track who experienced which.
7. **Settle** — costs and any creator payouts flow through OrangeCat.

A credible **MVP is just steps 1–4 for two modalities** (e.g. image + short text). Variants (5–6) and economy (7) are fast-followers once the loop feels magical.

## 6. The ecosystem participates in itself

The products shouldn't just share a logo family — **each should be a real user of the others**, and the system should be able to _simulate_ that usage to develop itself and to seed the communication and trust patterns that make people want to take part.

### (a) Mutual participation (composability)

Every product is simultaneously a **service** and a **customer**:

- FleetCrown pays for compute/identity via **OrangeCat**; OrangeCat runs its own agent fleet on **FleetCrown**.
- **Genesis** generates on **FleetRunner**, settles rights/payouts on **OrangeCat**, and is experienced through **FleetCrown**.
- **Flecoin** is the medium of exchange that flows between all of them.

OrangeCat already models this for real: FleetCrown is a typed "customer" with a stakeholder relationship and shared BTC wallet. Generalize that one relationship into the **default architecture** — every product holds an account, a wallet, and a reputation in every other product.

### (b) Synthetic economy (simulated usage)

Run **agents that play realistic users** — generating wishes, transacting, funding, reviewing, governing — to:

- exercise and harden the platforms before/alongside real users (load, edge cases, economic dynamics),
- bootstrap a living economy so the first real users arrive to a place that already feels alive,
- discover what interaction and trust patterns actually produce good outcomes.

**Honesty guardrail (non-negotiable):** synthetic participants must be _transparently synthetic_. They are a development and seeding tool, never passed off as real humans, never used to fake engagement, inflate markets, or manipulate real users' decisions. The moment simulated activity is presented as genuine human activity, it's deception — and it poisons the very trust the system is trying to build. Label synthetic accounts in-protocol; keep real and synthetic balances/markets distinguishable.

### (c) Communication & trust, aligned to what each person values

The connective layer is a **trust / reputation graph**: track record, attestations, fulfilled commitments — earned through real interactions (and stress-tested by the synthetic ones). The aim is communication that is _conducive to good and joyful outcomes_ and that surfaces what **each individual actually values** — so the system routes people toward experiences, collaborators, and exchanges aligned with their own goals rather than toward maximizing engagement. This is where OrangeCat's identity + governance layer and Genesis's "what do you wish for" signal reinforce each other: the wish reveals the value; the trust graph honors it.

## 7. Creators and the two-sided market

The engine isn't only a private wish-granter — it's the supply side of a **creator economy**. Each person runs an ongoing generative practice and an audience grows around it.

### (a) A personal generative practice

Every person can generate their own art — **a daily cadence if they want it**. Over time this becomes a body of work: a stream/feed of original pieces with clear provenance (it's _their_ output, so rights are clean — the §3 originality model is the foundation that makes everything below legal and resellable).

### (b) A following

Others **follow** creators whose work resonates. Following is two things at once:

- an **audience** (distribution + demand), and
- a **trust/value signal** that feeds the §6c trust graph — who you follow is a strong statement of what you value.

### (c) Sell it in any form

Any output is saleable across formats, the medium chosen by creator and buyer:

- **Digital** — files, licenses, limited editions, streams, access to a variant tree.
- **Physical** — prints, books, **vinyl/records**, fabricated objects, merch — via print-on-demand / fulfillment partners.
- **Across modalities** — visual art, **music**, **film/records**, 3D/AR pieces.

OrangeCat carries payments, creator payouts, and the rights/provenance that lets **royalties follow a work** (including across variants and resales); **Flecoin** is the unit of exchange. The follow graph drives discovery; the trust graph keeps it honest.

This closes the loop: **Genesis generates (supply) → following + marketplace create demand → OrangeCat settles value → trust graph aligns it to what people actually value.**

### Honest flags

- **Physical fulfillment is heavy operations** (manufacturing, shipping, returns, per-format partners). Treat digital-first as v1; physical (prints → records) as a deliberate phase-2 with fulfillment partners, not a launch feature.
- **Discovery/curation at daily-generation scale** is itself a hard problem — when everyone can make art every day, _finding the good_ is the bottleneck, not making it. The follow + trust graph is the proposed answer; it needs design.

## 8. Open questions (need answers before building)

1. **Is Genesis a new product, or a feature inside OrangeCat / FleetCrown?** Affects repo layout, branding, auth.
2. **Which modalities for v1?** Recommend the cheapest-to-delight pair first (image + text) before video/3D/VR (orders of magnitude more cost/latency).
3. **Which model providers?** Per-modality routing means several. Need a provider matrix + budget per wish.
4. **Where does generation run?** FleetRunner local (own keys/compute) vs. cloud workers. Local is the stated intent; confirm.
5. **What is the unit of value in OrangeCat?** Pay-per-generation, pay-per-experience, subscription, creator royalties on variants — or a mix.
6. **Distribution model for variants** — random assignment, chosen by audience, personalized, or experimental cohorts?
7. **Rights/provenance model** — how is ownership of an original world/character represented so variants and payouts attach to it?
8. **Is the synthetic economy internal (dev/testing only) or a public-facing mechanic?** If public, the transparency/disclosure requirements (§6b) become product surface, not just policy.
9. **What is the trust graph's substrate** — reputation native to OrangeCat, attestations, or an existing standard — and how is "what a person values" captured without becoming surveillance?
10. **Marketplace model** — digital-first for v1 (which formats: files, licenses, editions, streams)? When/how do physical formats (prints → records) enter, and via which fulfillment partners?
11. **Discovery & curation** at daily-generation scale — how does the follow + trust graph surface quality without collapsing into an engagement-maximizing feed?
12. **Royalty/edition model** — limited editions, resale royalties, variant-tree royalties: what's the default that's fair to creators and simple to explain?

## 9. Suggested first step

Don't build across three repos at once. Prove the magic in one place first:

- A thin **"wish → image + caption" vertical slice** in FleetCrown (web), calling out to one image model and one text model, returning a finished card.
- If that loop feels like magic, lift the orchestration into **FleetRunner** and add a second modality.
- Add **variants** (same wish, N deliberate deltas) on original outputs.
- Wire **OrangeCat** for cost/payout last.
